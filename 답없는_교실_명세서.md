# No-dap-class — 전체 개발 명세서 (최종 확정)

> "AI가 답을 주는 걸 구조적으로 막는 학습 파트너"
>
> 바이브코딩으로 만들되, 발표 때 모든 코드의 동작 원리를 설명할 수 있어야 한다.

---

## 1. 프로젝트 개요

### MVP 기능 흐름 (최종 확정)

```
1. 사용자가 코드/기획안/주장/학습내용을 텍스트로 입력
   또는 이미지/텍스트 파일로 업로드
2. AI가 읽는다
3. AI가 헛점, 질문, 이해 확인 등을 바탕으로 말한다 (답은 절대 주지 않음)
4. 사용자가 대답한다
5. 3~4를 반복 (최소 5턴 강제)
6. 5턴 달성 시 종료 버튼 활성화
7. AI가 "지금까지 대화를 바탕으로 본인이 이해한 내용을 직접 정리해보세요"로 유도
8. 사용자가 최종 정리본 직접 작성 → 이게 최종 결과물
9. 결과 화면: 좌측 = 사용자 최종 정리본 / 우측 = AI 분석 정리집 (반반 레이아웃)
10. 하단: 전체 대화 과정 아코디언 (접기/펼치기)
```

### 핵심 제약 (제품 철학)
- AI는 절대 코드/답을 먼저 주지 않는다
- 시스템이 이를 구조적으로 강제한다 (프롬프트 레벨 + UI 레벨 동시)
- 최소 5턴 이상 대화해야 종료 버튼이 활성화된다
- PDF export 없음 — 화면 표시만 (MVP 이후 추가 가능)
- 로그인/로그아웃 없음 — 데이터는 프론트 localStorage에만 저장
- 백엔드는 완전 stateless — 매 요청마다 프론트가 전체 세션 데이터 전송
- 세션 이어하기 미구현 → 데모는 항상 새 탭에서 시작

## 2. 기술 스택

| 영역 | 선택 | 이유 |
|------|------|------|
| Frontend | React + Vite (TypeScript) | 팀 경험 있음, 빠른 개발 |
| Styling | TailwindCSS | 빠른 UI 구성 |
| HTTP 클라이언트 | axios | API 호출 |
| 라우팅 | react-router-dom | 페이지 전환 |
| Backend | Express.js (Node.js + TypeScript) | 팀 경험 있음, JS 언어 통일 |
| API 문서 | Swagger (swagger-jsdoc + swagger-ui-express) | API 명세 자동화 |
| AI 모델 | GPT-4o | 이미지 분석 지원, 한국어 품질 |
| 파일 업로드 | multer | 이미지/텍스트 파일 처리 |
| 저장 | 프론트 localStorage (백엔드 stateless) | DB 없이 세션 유지, 개인정보 수집 없음 |
| 배포 | 미정 | MVP 이후 결정 |

// frontend/src/types/index.ts

export type Role = 'user' | 'assistant'

  title: string
  initial_input: string
  messages: Message[]
  turn_count: number
  user_final_output: string
  created_at: string
  completed_at?: string
}

export interface AISummary {
  understood: string[]
  lacking: string[]
  key_concepts: string[]
}

## 5. API 명세서

### Base URL
```
http://localhost:4000
```

---

### 5-1. POST /api/sessions — 세션 시작

**Request Body**
```json
{
  "initial_input": "웹개발할 때 알고리즘을 배워야할까요? 나는 필요없다고 생각해요"
}
```

**Response**
```json
{
  "session_id": "uuid-v4",
  "title": "웹개발할 때 알고리즘을 배워야",
  "created_at": "2025-01-01T00:00:00Z"
}
```

---

### 5-2. POST /api/sessions/{session_id}/chat — 대화 (SSE 스트리밍)

**Request Body**
```json
{
  "initial_input": "처음 올린 내용 전문",
  "messages": [
    { "role": "user", "content": "라이브러리가 다 해주니까 필요없어요" }
  ],
  "turn_count": 1
}
```

**Response (SSE 스트림)**
```
data: {"type":"text","value":"[분석]:"}
data: {"type":"text","value":" 흥미로운 관점이네요..."}
data: [DONE]
```

> turn_count >= 5이면 AI 응답 말미에 최종 정리 유도 멘트 자동 추가

---

### 5-3. POST /api/sessions/{session_id}/summary — AI 분석 정리집 생성

**Request Body**
```json
{
  "initial_input": "처음 올린 내용 전문",
  "messages": [
    { "role": "user", "content": "..." },
    { "role": "assistant", "content": "..." }
  ]
}
```

**Response**
```json
{
  "understood": ["라이브러리 활용으로 대부분의 작업을 수행할 수 있다고 생각함"],
  "lacking": ["알고리즘의 중요성을 이해하지 못함"],
  "key_concepts": ["알고리즘 기본 개념", "웹 개발과 알고리즘의 관계", "라이브러리 사용과 알고리즘의 차이"]
}
```

---

### 5-4. POST /api/sessions/{session_id}/finish — 최종 정리본 제출

**Request Body**
```json
{
  "user_final_output": "사용자가 직접 작성한 최종 정리본 텍스트"
}
```

**Response**
```json
{
  "completed_at": "2025-01-01T00:00:00Z",
  "user_final_output": "사용자가 직접 작성한 최종 정리본 텍스트"
}
```

---

### 5-5. POST /api/sessions/{session_id}/image — 이미지 업로드 및 분석

**Request**: multipart/form-data
- `image`: 이미지 파일 (jpg, png 등)
- `user_text`: 이미지와 함께 전달할 텍스트 (선택)

**Response**
```json
{
  "ai_response": "[이미지 분석]: 이미지에서 보이는 내용...\n[질문]: 핵심 질문?"
}
```

> 프론트에서 `ai_response`를 받아 `initial_input`으로 세팅 후 `/chat` 호출

---

### 5-6. POST /api/sessions/{session_id}/file — 텍스트 파일 업로드

**Request**: multipart/form-data
- `file`: 텍스트 파일 (.txt, .md, .json, .html, .css, .js)

**Response**
```json
{
  "extracted_text": "파일에서 추출된 텍스트 내용"
}
```

> 프론트에서 `extracted_text`를 받아 `initial_input`으로 세팅 후 `/chat` 호출

## 6. 화면 구성 (최종 확정)

### 6-1. 랜딩 페이지 (/)

- 좌측: 안내 문구 + 이미지/파일 업로드 영역
- 우측: 텍스트 직접 입력 textarea
- 하단: START SESSION 버튼
- 입력값을 localStorage에 저장 후 /session으로 이동

### 6-2. 메인 학습 화면 (/session)

3-Panel 레이아웃 (좌 / 중 / 우)

| 패널 | 컴포넌트 | 역할 |
|------|----------|------|
| 좌측 | InputPanel | 처음 제출한 내용 읽기 전용 표시 |
| 중앙 | ChatPanel | SSE 스트리밍 대화, 사용자 입력 textarea, 제출 버튼 |
| 우측 | ProgressPanel | 턴 카운터 (N/5), 프로그레스바, 종료 버튼 |

**종료 버튼 동작 흐름**
1. 5턴 전: 종료 버튼 비활성 (회색)
2. 5턴 달성: 종료 버튼 활성화
3. 종료 버튼 클릭 → ChatPanel 하단에 최종 정리 textarea 노출
4. 사용자가 최종 정리본 작성
5. "결과 보기" 버튼 → session 전체 localStorage 저장 → /result 이동

### 6-3. 결과 화면 (/result)

반반 레이아웃 (좌 / 우) + 하단 아코디언

| 영역 | 내용 |
|------|------|
| 좌측 (50%) | 내가 정리한 것 — 사용자가 직접 작성한 최종 정리본 |
| 우측 (50%) | AI 분석 — 이해한 것 / 아직 부족한 것 / 핵심 개념 |
| 하단 | 전체 대화 과정 아코디언 (기본 닫힘) |

**결과 화면 동작**
- 진입 시 localStorage에서 session 불러옴
- 동시에 POST /api/sessions/{id}/summary 호출
- AI 분석 로딩 중 → 우측 스켈레톤 UI 표시
- 아코디언 클릭 시 전체 대화 내용 펼쳐짐

## 7. 프론트 데이터 흐름

```
이미지/파일 업로드 시:
/image 또는 /file 호출 → ai_response / extracted_text 수신
→ initial_input으로 세팅 → localStorage 저장 → /session 이동

텍스트 직접 입력 시:
textarea 입력 → /sessions 호출 → session_id 수신
→ localStorage 저장 → /session 이동

대화 진행 시:
사용자 입력 → /chat 호출 (initial_input + messages + turn_count 전송)
→ SSE 스트리밍 수신 → messages 업데이트 → localStorage 저장

종료 시:
/finish 호출 → /result 이동
→ /summary 호출 → AI 분석 결과 화면에 렌더링
```

## 8. localStorage 유틸

```typescript
// frontend/src/lib/storage.ts

import { Session } from '../types'

const SESSION_KEY = 'nodap_session'

export const saveSession = (session: Session): void => {
  localStorage.setItem(SESSION_KEY, JSON.stringify(session))
}

export const loadSession = (): Session | null => {
  const data = localStorage.getItem(SESSION_KEY)
  return data ? JSON.parse(data) : null
}

export const clearSession = (): void => {
  localStorage.removeItem(SESSION_KEY)
}
```

## 12. 발표 때 반드시 설명할 수 있어야 하는 것

- SSE 스트리밍이 왜 필요한가 (일반 fetch와 뭐가 다른가)
- 시스템 프롬프트가 어떻게 AI 동작을 구조적으로 강제하는가
- turn_count 조건 분기로 유도 멘트를 어떻게 주입하는가
- 왜 백엔드를 stateless로 설계했는가
- 왜 DB 없이 localStorage를 선택했는가 (개인정보 수집 없음)
- 이미지/파일 업로드 후 어떻게 대화로 연결되는가
- 2차 API 호출(summary)이 어떤 구조로 동작하는가
- 5턴 잠금 로직이 왜 제품 철학과 연결되는가
- 서비스 파일을 chat/summary/image로 분리한 이유
